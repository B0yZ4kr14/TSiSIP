## 1. Purpose

This document defines the governance framework for durable agent memory within the TSiSIP project. It ensures that AI agents retain high-signal context across sessions without compromising security, privacy, or architectural integrity.

---

## 2. Memory Hierarchy

```text
L1 — Constitution      .specify/memory/constitution.md          (immutable rules)
L2 — Architecture      .specify/memory/architecture_constitution.md  (topology, stack)
L3 — Decisions         docs/memory/DECISIONS.md                 (ADRs, rejected patterns)
L4 — Bugs              docs/memory/BUGS.md                      (resolved + active)
L5 — Worklog           docs/memory/WORKLOG.md                   (session history)
L6 — Synthesis         docs/memory/memory-synthesis.md          (condensed context)
L7 — Feature Memory    specs/NNN-*/memory.md                    (per-feature context)
```

**Access Rules**:
- L1–L2: Read-only for agents. Human-only writes via Architecture Review.
- L3–L5: Agent-readable. Writes require explicit human approval via memory-md.capture.
- L6: Auto-generated by memory-md.audit. Human review before commit.
- L7: Feature-scoped. Created during speckit-specify and updated during implementation.

---

## 3. What May Be Remembered

**Allowed**:
- Architecture decisions and ADRs
- Rejected patterns and anti-patterns
- Bug root causes and fixes
- Build/test commands and their outputs
- Docker/Compose network topology
- OpenSIPS routing logic conventions
- PostgreSQL schema conventions
- Security control implementations (without secrets)

**Forbidden**:
- Passwords, API keys, tokens, salts
- Private keys or certificate material
- HA1 hashes or pre-computed auth values
- Production IP addresses or hostnames
- Tenant-specific routing data
- PII from CDRs or logs
- Unredacted stack traces with file paths outside repo

---

## 4. Capture Workflow

```
Agent completes task / diff
  -> memory-md.capture proposes update
    -> Human review (yes/no/edit)
      -> Commit to docs/memory/
```

### 4.1 Approval Criteria
Before approving a memory capture, verify:
1. No secrets or credentials are present
2. No PII or production data is included
3. The lesson is generalizable (not a one-off hack)
4. The information is durable (still relevant in 30+ days)
5. Source attribution is included (feature ID, commit hash)

### 4.2 Rejection Reasons
Capture MUST be rejected if:
- Contains any pattern from .gitignore or secrets/
- Includes environment-specific values (IPs, ports outside canonical spec)
- Duplicates existing memory without adding new context
- Is speculative rather than factual

## 6. Memory Quality Gates

Before each feature implementation, agents MUST:
1. Read memory-synthesis.md for condensed context
2. Read DECISIONS.md for constraints on current problem
3. Read BUGS.md for known pitfalls
4. Check for conflicts between proposed solution and existing decisions

After each feature completion, agents MUST:
1. Propose captures for durable lessons
2. Update WORKLOG.md with session summary
3. Run memory-md.audit to check index integrity

---

## 7. Conflict Resolution

When memory contradicts current implementation:
1. Memory wins if it is a Constitution (L1–L2) item.
2. Implementation wins if memory is a stale Worklog (L5) entry.
3. Escalate to Architecture Review if L3 (Decision) conflicts with L3 (newer Decision).

Resolution is logged in DECISIONS.md with:
- Conflicting entries (with dates)
- Resolution rationale
- Approver name
- Feature ID that triggered the conflict

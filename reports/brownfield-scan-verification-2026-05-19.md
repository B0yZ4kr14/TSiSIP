## Remaining Findings

### Finding 1: Stale Dispatcher Comment (`opensips.cfg.tpl`)

- **Location:** `opensips.cfg.tpl:213`
- **Severity:** Low
- **Issue:** The inline comment in the trunk-integration documentation block states that `ds_probing_threshold=5` applies. However, the actual `modparam` at line 113 sets the value to `2`.
- **Impact:** Operator confusion during incident response or configuration audits. The comment predates the B6 fix and was not updated.
- **Recommendation:** Update the comment to read `Global ds_probing_threshold=2 applies;`.

### Finding 2: RTPengine Control Socket Binds to `0.0.0.0` in Dev Stack (`docker-compose.yml`)

- **Location:** `docker-compose.yml:152`
- **Severity:** Low
- **Issue:** The development stack binds the RTPengine NG control socket to `0.0.0.0` via the command argument `--listen-ng=0.0.0.0:22222`. The canonical specification (`AGENTS.md` §10, rejected patterns) mandates binding to `${RTPENGINE_INTERNAL_IP}:22222` on `sip_internal`. Both `docker-compose.prod.yml:67` and `docker-compose.vps.yml:106` correctly use `${RTPENGINE_INTERNAL_IP}:22222`; only the dev profile deviates.
- **Impact:** Slightly wider exposure surface within the `sip_internal` Docker network. While the port is not published to the host, any container on `sip_internal` can reach the control socket, which violates the principle of least privilege.
- **Recommendation:** Align the dev profile with production by changing the argument to `--listen-ng=${RTPENGINE_INTERNAL_IP}:22222` and ensuring the `RTPENGINE_INTERNAL_IP` environment variable is defined for the service.

# Feature 032: Automated Backup Verification & Disaster Recovery Testing

## Overview

| Field | Value |
|-------|-------|
| **Feature** | Automated Backup Verification & Disaster Recovery Testing |
| **Short name** | automated-backup-verification |
| **Created** | 2026-05-27 |
| **Status** | In Progress |
| **Last Updated** | 2026-05-27 |
| **Context** | The existing backup script (`scripts/backup-db.sh`) performs basic gzip integrity checks but does not verify that the PostgreSQL dump can actually be restored. There is no automated disaster recovery validation, no backup checksum integrity tracking, and no alerting when backups fail verification. This feature adds comprehensive backup verification, automated restore testing, and disaster recovery runbook automation. |
| **Objective** | Ensure every database backup is not only created but verified restorable; automate periodic disaster recovery drills; provide operators with clear evidence of backup health and recovery time objectives (RTO). |

## Goals

1. **Restore Verification**: After each backup, automatically restore it to a temporary PostgreSQL container and verify schema integrity, row counts, and critical table presence.
2. **Checksum Integrity**: Generate and store SHA-256 checksums for each backup file; verify on restore to detect corruption.
3. **Backup Health Dashboard**: Add an OCP admin page showing backup history, verification status, storage usage, and retention compliance.
4. **Automated DR Drills**: Schedule monthly automated disaster recovery tests that restore the latest backup to an isolated environment and run a smoke test suite.
5. **Alerting on Failure**: Integrate with existing Prometheus/Alertmanager to fire alerts when backup verification fails or DR drill RTO exceeds threshold.
6. **Encrypted Backup Verification**: Verify that encrypted backups (for S3 upload) can be decrypted and restored successfully.

## Non-Goals

- Real-time backup replication (already handled by Feature 005).
- Cross-region backup synchronization (deferred to future feature).
- Application-level backup (Asterisk voicemail, recordings; deferred).

## Acceptance Criteria

- [ ] **AC1**: Every backup generated by `scripts/backup-db.sh` has a corresponding `.sha256` checksum file.
- [ ] **AC2**: A new `scripts/verify-backup.sh` script restores the latest backup to a temporary container and validates schema + row counts.
- [ ] **AC3**: OCP admin page `/backup-status.php` displays backup history with verification status (PASS/FAIL).
- [ ] **AC4**: Prometheus metric `tsisip_backup_last_verify_timestamp` and `tsisip_backup_verify_success` are exposed.
- [ ] **AC5**: Alertmanager rule fires when `tsisip_backup_verify_success == 0` for > 5 minutes.
- [ ] **AC6**: Monthly DR drill script `scripts/dr-drill.sh` runs via cron and produces an evidence report.
- [ ] **AC7**: All encrypted backups are verified by decrypting before restore test.

# /speckit.team.assign

## Purpose

Read the current feature's `tasks.md` and `team.yml`, assign each top-level task to an engineer based on role and capacity, and optionally split complex tasks into subtasks. Write the result back to `tasks.md` with assignment annotations.

## Instructions

### Step 1 — Load required files

Read the following files. If any are missing, stop and tell the user which file is missing and how to create it:

- `.specify/specs/<feature>/tasks.md` — the task list generated by `/speckit.tasks`
- `.specify/team.yml` — the team configuration created by `/speckit.team.setup`

If there are multiple feature directories, ask the user which feature to assign.

### Step 2 — Parse the task list

Identify every top-level task in `tasks.md`. Tasks follow this pattern:

```
- [ ] Task title
  - Additional context or notes
```

For each task, determine:
- **Domain**: is this a backend, frontend, devops, qa, or general task? Infer from the task title and context.
- **Complexity**: estimate simple (1 point), medium (2–3 points), or complex (4+ points) based on scope described.
- **Dependencies**: note if the task explicitly depends on another task.

### Step 3 — Assign tasks to engineers

Apply the strategy from `team.yml`:

**role-match** (default):
- Match each task's domain to the engineer whose `role` best fits.
- Respect capacity: do not exceed any engineer's capacity limit.
- If a role has no available capacity, assign to the nearest role or flag as unassigned.

**round-robin**:
- Distribute tasks in order across all engineers.
- Respect capacity limits.

**manual**:
- Do not assign anyone. Leave an empty `[ @___ ]` placeholder on every task.
- Print a suggestion in a comment after each task line showing who *would* be assigned under role-match.

### Step 4 — Split complex tasks into subtasks

For any task estimated as **complex** (4+ points):

1. Analyse the task description and spec context.
2. Propose a breakdown into 2–4 concrete subtasks, each completable independently.
3. Ask the user: *"Task X looks complex. I suggest splitting it into these subtasks — shall I apply this?"*
4. If confirmed, replace the single task line with a parent task and indented subtask list:

```
- [ ] Parent task title [@alice] [3pts]
  - [ ] Subtask 1 — brief description
  - [ ] Subtask 2 — brief description
  - [ ] Subtask 3 — brief description
```

If the user declines, keep the task as a single line and assign normally.

### Step 5 — Write the updated tasks.md

Rewrite `tasks.md` in place with the following additions for each task:

```
- [ ] Task title [@assignee] [Npts]
```

- `[@assignee]` is the engineer's name or GitHub handle from `team.yml`.
- `[Npts]` is the estimated complexity in points.
- Unassigned tasks get `[@unassigned]`.

Add a **Team Summary** section at the top of `tasks.md` below the header:

```markdown
## Team summary — Sprint 1

| Engineer | Role       | Assigned | Capacity | Status  |
|----------|------------|----------|----------|---------|
| Alice    | backend    | 4 pts    | 5 pts    | ✅ OK   |
| Bob      | frontend   | 5 pts    | 5 pts    | ✅ OK   |
| Carol    | fullstack  | 3 pts    | 4 pts    | ✅ OK   |
| Dave     | devops     | 2 pts    | 3 pts    | ✅ OK   |

> Generated by /speckit.team.assign
```

Use ⚠️ Over capacity if an engineer is assigned more than their capacity.

### Step 6 — Report to the user

After writing the file, print a brief report:

```
Tasks assigned ✓

  12 tasks across 4 engineers
  2 tasks split into subtasks (Task A, Task B)
  0 tasks unassigned

Run /speckit.team.board to see the full workboard view.
```

If any tasks could not be assigned (no matching role, over capacity), list them explicitly and ask the user to reassign them manually or adjust capacity in `team.yml`.

## Notes

- Never remove existing task content — only append assignment and point annotations.
- If `tasks.md` already has assignments (re-run scenario), ask the user: "tasks.md already has assignments. Reassign from scratch or update only unassigned tasks?"
- Respect the `[P]` parallel marker that `/speckit.tasks` may have added; do not remove it.
- Point estimates are suggestions — the user can override any estimate by editing `tasks.md` directly.

# TSiSIP Post-Implementation Critique Report

> **Date:** 2026-05-26
> **Scope:** Overall project state (Features 001–024, production VPS stack, 10-stage roadmap)
> **Critique Type:** Dual-lens review — Product Strategy + Engineering Risk

---

## Executive Summary

TSiSIP is a **well-architected, constitution-compliant SIP edge proxy platform** that has successfully delivered 23+ feature specifications and achieved a stable VPS production deployment. The foundational architecture (Docker-first, PostgreSQL-only, OpenSIPS 3.6 LTS, three-network topology) is sound and consistently enforced. Brownfield remediation is complete (16/16 findings resolved), and the OCP admin tooling now provides genuine operational value through 6 critical gap-closure tools.

However, the project sits at a **pivotal inflection point**: the SIP edge is technically deployed but **not yet handling public traffic** due to an upstream ACL/NAT block outside the host. The 10-stage roadmap is logical but front-loads internal hygiene (schema, VEX, supply chain) while deferring the product-defining milestone — public SIP exposure — to Stage 6. Several engineering gaps (floating `:latest` tags, missing monitoring on VPS, unbounded audit log growth, no MI HTTP circuit breaker) create operational fragility that will compound under real traffic.

**Verdict:** ⚠️ **PROCEED WITH UPDATES** — The architecture is solid and the team should proceed, but three 🎯 Must-Address items must be resolved before public SIP exposure, and the roadmap sequencing warrants reconsideration.

## Product Lens Findings

### 3a. Problem Validation

**P1 — Roadmap front-loads hygiene over product validation** 🎯

The 10-stage roadmap sequences schema hardening, VEX generation, and supply chain determinism (Stages 1–3) before public SIP exposure (Stage 6). While these are important, the project cannot validate its core value proposition — "the only public SIP signaling entry point" — until Stage 6. If the SIP edge has undiscovered failure modes under real traffic, 6+ weeks of investment in deterministic image tagging may need re-evaluation.

> **Suggestion:** Parallelize Stage 6 (public SIP exposure) with Stages 1–3. Use a staging environment or limited beta whitelist to validate call flows while internal hardening proceeds. SIP exposure is the primary product risk; it should not be deferred behind CI artifacts.

**P2 — Missing Feature 014 creates ambiguity** 🤔

The spec directory jumps from `013` to `015`, leaving Feature 014 absent. AGENTS.md references Features 001–023, then 024, but the canonical spec references Feature 015 as "Auto TLS Certificate Rotation" (consistent with the directory). The gap creates uncertainty about whether 014 was intentionally skipped, merged, or overlooked.

> **Suggestion:** Document the absence of Feature 014 explicitly in `STATUS.md` or `ROADMAP.md` (e.g., "Feature 014 was intentionally reserved/merged into Feature 015").

### 3b. User Value Assessment

**P3 — OCP admin tools (Feature 020) deliver real value but have UX blind spots** 💡

Feature 020 successfully closed 6 critical tool gaps (Dialog Viewer, MI Commands, Statistics Monitor, Dialplan Manager, Domains Manager, TLS Management). These tools are genuinely useful for operations. However, three UX/product gaps remain:

1. **D3.js CDN dependency**: `statistics.php` loads D3.js from a CDN; air-gapped deployments degrade gracefully but lose charts entirely.
2. **MI HTTP timeout cascading**: No circuit breaker means that if the MI HTTP endpoint fails, both `statistics.php` and `mi-commands.php` fail simultaneously — a poor operator experience during incidents.
3. **Statistics auto-refresh load**: Fixed 30-second polling generates continuous MI HTTP requests (2 req/min per active tab) with no user toggle or backoff.

> **Suggestion:** Vendor D3.js into `web/tsisip/js/` for air-gapped support. Add a shared MI HTTP health/circuit breaker state. Replace fixed-interval polling with adaptive refresh or a manual refresh button.

**P4 — Default admin credential in seed data is a product security anti-pattern** 🎯

The seed data still contains a hardcoded default admin credential (documented in the runbook). While `force_password_change = TRUE` is set, the credential exists in the database initialization SQL and is documented in the runbook. This contradicts the constitution's "no plaintext secrets" principle and creates a window of vulnerability on every fresh deployment.

> **Suggestion:** Generate the initial admin passphrase via the entrypoint script at first boot, write it once to a Docker secret, and print it to container logs. Remove the hardcoded credential from seed data entirely.

**P5 — SIP trunk integration (Feature 017) is implemented but not production-exercised** 💡

Feature 017 implements SIP trunk schemas, outbound/inbound routing, rate limiting, and health probes. However, the trunk integration has not been validated with real provider traffic because public SIP exposure is blocked upstream. The DID management UI and trunk health dashboard are deferred to Stage 7. Without real traffic, failover behavior, provider latency, and rate-limit accuracy are theoretical.

> **Suggestion:** After public SIP exposure (Stage 6), prioritize a trunk provider pilot with a single DID before building the full DID management UI. Real provider behavior (especially around REGISTER refresh intervals and 487 handling) often diverges from spec assumptions.

### 3c. Alternative Approaches

**P6 — VPS-lite profile disables the full observability stack** 💡

The production VPS runs 10 services with Prometheus, Grafana, Alertmanager, and anomaly-detector disabled due to memory constraints (~7.5GB total). This means the production environment operates without the monitoring, alerting, and anomaly detection that the project invested heavily in (Features 003, 004, 006). An alternative approach would be to run a lightweight metrics/alerting sidecar or external SaaS monitoring instead of the full Prometheus/Grafana stack.

> **Suggestion:** Evaluate a lightweight alternative for the VPS-lite profile — e.g., a single `node_exporter` + `opensips_exporter` pushing to an external Prometheus, or a minimal Alertmanager-only container for critical alerts. The current "all or nothing" approach leaves production blind.

### 3d. Edge Cases & User Experience

**P7 — Audit log table growth is unbounded** 🎯

All 6 new OCP tools plus remediation logging write to `auth_audit_log`. The runbook documents queries but no retention, partitioning, or purge strategy. Under real multi-tenant traffic, this table will grow indefinitely and degrade OCP query performance.

> **Suggestion:** Define a retention policy (e.g., 90 days for auth events, 365 days for CDRs) and implement automated purge or PostgreSQL table partitioning by month. Add `web/cli/purge-audit-log.php` to the cron pipeline.

**P8 — No documented rollback for schema migrations** 💡

The operator runbook documents full-stack rollback via `git checkout`, but there is no guidance for rolling back a failed PostgreSQL schema migration (e.g., the Stage 1 `tenant_id` type migration). In a production environment with live subscribers, a failed migration is a P0 incident.

> **Suggestion:** Add a "Schema Migration Rollback" section to the runbook with `pg_dump` pre-migration snapshot procedures and `ALTER TABLE` reversal commands for each supported migration.

### 3e. Success Measurement

**P9 — No defined SLIs/SLOs for SIP edge availability** 💡

The project has excellent acceptance criteria (AC-01 through AC-17 in the canonical spec) but no defined service-level indicators for production. What is the target SIP registration success rate? Target call setup time? Target RTP media quality (MOS)? These metrics are necessary to evaluate whether the platform is "successful" after public exposure.

> **Suggestion:** Define initial SLIs in the runbook: e.g., "SIP registration success rate > 99.5%", "INVITE setup time < 2s p99", "RTP packet loss < 0.1%". Instrument these via the Prometheus exporter or CDR analysis.

## Engineering Lens Findings

### 4a. Architecture Soundness

**E1 — Architecture is constitution-compliant and well-isolated** ✅

The three-network topology (`sip_edge`, `sip_internal`, `db_internal`) is consistently enforced. Asterisk and PostgreSQL have zero host-published ports. OpenSIPS 3.6 LTS is the sole SIP engine. PostgreSQL is the only database. The auth contract uses precomputed HA1 (`calculate_ha1 = 0`). Topology hiding uses `topology_hiding("C")`. These are all strong architectural decisions.

> **No action required.**

**E2 — Schema inconsistency in `sip_trunk_did_mappings`** 🎯

The `sip_trunk_did_mappings` table uses `UUID` for `tenant_id` while the rest of the schema (including `subscriber`, `header_routing_rules`, `pbx_backends`) uses `VARCHAR(36)` for tenant IDs. This inconsistency breaks the project's own schema contract and could cause join failures or index mismatches.

> **Suggestion:** Execute Stage 1 deliverable 1.2 immediately — migrate `tenant_id` to `VARCHAR(36)` and add the schema regression test before any other Stage 1 work.

**E3 — Floating `:latest` tags in production Compose files** 🎯

`docker-compose.vps.yml` and `docker-compose.prod.yml` use `:latest` tags for GHCR images. This violates the constitution ("SHA256-digest-pinned base images") and the Stage 3 roadmap goal. A bad push to `:latest` could auto-deploy a broken image on the next `docker compose pull`.

> **Suggestion:** As an interim measure before full Stage 3 implementation, pin the VPS compose file to the currently known-good image digests (recorded in the deploy pipeline's pre-deploy capture). Then implement `make release-tag`.

### 4b. Failure Mode Analysis

**E4 — No documented failure mode for PostgreSQL primary failure** 💡

The project has a single PostgreSQL container with backup, WAL archiving, and PITR — but no high-availability or automatic failover path. If the PostgreSQL container crashes, OpenSIPS loses auth and routing metadata immediately. The dispatcher failover handles Asterisk backend failures, but not the database failure.

> **Suggestion:** Document the PostgreSQL failure mode explicitly: what happens to in-flight calls (they continue via dialog state), what happens to new registrations (they fail), and the expected RTO for restore from backup. Evaluate `pg_auto_failover` or Patroni for a future HA iteration.

**E5 — OpenSIPS `pkg_mem_size` may be tight under high load** 💡

Memorylint finding M1 notes that `pkg_mem_size` may be tight under high load, but no explicit value is set in `opensips.cfg.tpl` (M5). With 10x growth, package memory exhaustion could cause OpenSIPS worker crashes.

> **Suggestion:** Add explicit `shm_mem_size` and `pkg_mem_size` to `opensips.cfg.tpl` with values sized for the expected concurrent call volume. Document the sizing formula in the runbook.

**E6 — No circuit breaker for MI HTTP dependency** 💡

AGENTS.md documents a known issue: "If the MI HTTP endpoint becomes unreachable, statistics.php and mi-commands.php both enter error states simultaneously. No circuit breaker exists." This is a cascading failure pattern.

> **Suggestion:** Implement a simple circuit breaker in PHP: track MI HTTP failures in `$_SESSION` or APCu; after 3 consecutive failures, return a cached "Service temporarily unavailable" response for 60 seconds instead of blocking on the timeout.

### 4c. Security & Privacy Review

**S1 — Security model is comprehensive but untested under real attack** 💡

The security model covers SIP ingress auth, header sanitization, credential stripping, topology hiding, TLS/SRTP, Docker capability dropping, and LGPD compliance framework. Brownfield remediation resolved all 16 findings. However, the rate limiting and DDoS protection (Feature 006) is only "partial live" — the anomaly detector is disabled on the VPS, and external flood testing is pending.

> **Suggestion:** Before public SIP exposure, conduct a controlled penetration test or SIP flood simulation against the VPS edge (from an external host) to validate pike, ratelimit, and userblacklist behavior under real attack volume.

**S2 — `auth_audit_log` lacks retention; LGPD compliance deferred** 🎯

LGPD compliance automation is deferred to Stage 9 (weeks 9–12). However, the audit log already contains PII (username, source IP, call ID) with no retention policy. If the VPS handles real traffic before Stage 9, the project will be storing PII indefinitely without a compliant purge mechanism.

> **Suggestion:** Implement a minimal retention purge for `auth_audit_log` and `ocp_login_log` (e.g., 90 days) immediately, even before the full LGPD Stage 9. This is a legal/regulatory precondition for handling subscriber data.

### 4d. Performance & Scalability

**E7 — PostgreSQL `work_mem` × `max_connections` approaches reservation** 💡

Memorylint M2 flags that PostgreSQL memory configuration may approach the container reservation. The VPS allocates 2GB to PostgreSQL. Under 10x growth (more tenants, more concurrent calls, more CDR writes), this reservation could become a bottleneck.

> **Suggestion:** Document the current `work_mem`, `shared_buffers`, and `max_connections` values in the runbook. Add a connection pooler (e.g., PgBouncer) to the roadmap for Stage 7+ if concurrent connection count exceeds ~100.

**E8 — RTPengine userspace fallback vs. kernel module** 🤔

The project uses containerized RTPengine without kernel acceleration. `STATUS.md` notes: "RTPengine kernel table: Deferred — Decide between kernel module on host or extra memory for userspace fallback." At high call volume, userspace RTP relay consumes significantly more CPU.

> **Suggestion:** Benchmark RTPengine CPU usage under simulated load (e.g., 50 concurrent G.711 calls) and document the breakpoint where kernel acceleration becomes necessary. This affects VPS sizing decisions.

### 4e. Testing Strategy

**E9 — Integration tests exist but are not run in CI against the VPS** 💡

The project has an extensive integration test suite (15+ test files), but the CI workflow runs them against a local compose stack, not the actual VPS. The VPS has unique configuration (e.g., `userland-proxy=false`, nginx proxy, Cloudflare TLS) that local tests cannot validate.

> **Suggestion:** Add a post-deploy smoke test job to the GitHub Actions deploy workflow that runs a subset of integration tests (OPTIONS probe, INVITE auth, OCP health) against the live VPS endpoint using repository secrets for the target host.

**E10 — No load or soak tests** 💡

The test suite validates functional correctness but not performance under sustained load. There is no test for 100+ concurrent registrations, 50+ concurrent calls, or 24-hour stability.

> **Suggestion:** Add a `tests/load/` directory with SIPp scenarios for registration storms and call volume tests. Run these before public SIP exposure to validate memory and CPU headroom.

### 4f. Operational Readiness

**E11 — Monitoring stack disabled on production VPS** 🎯

Prometheus, Grafana, Alertmanager, and anomaly-detector are all disabled on the VPS-lite profile. The only operational telemetry available is the backup metrics exporter on the internal Docker network. This means the production environment has **no alerting** for OpenSIPS crashes, PostgreSQL failures, RTPengine overload, or certificate expiry (except via certbot-exporter, which is also internal-only).

> **Suggestion:** Re-enable the monitoring overlay as soon as memory/policy allows (Stage 5). As a critical interim measure, add a simple external health check ping (e.g., UptimeRobot or a cron job on a secondary host) that alerts if the SIP endpoint stops responding to OPTIONS.

**E12 — Backup cron jobs not yet observed** 💡

The backup cron schedules (02:00, 03:00, 04:00 UTC) are installed but the first automatic window execution has not been observed. `STATUS.md` notes: "missing first automatic window observation."

> **Suggestion:** After the next cron window, verify that `/backup/daily/`, `/backup/wal/`, and the metrics exporter all show expected state. Document the observation in `STATUS.md`.

**E13 — No documented certificate expiry alerting** 💡

The certbot container handles auto-renewal, but there is no documented alerting path if renewal fails. The certbot-exporter exists but is on an internal network with no external scraping.

> **Suggestion:** Add an external expiry monitor (e.g., a weekly cron on a secondary host that checks HTTPS expiry and emails on < 14 days). Alternatively, expose certbot-exporter metrics through the nginx proxy.

### 4g. Dependencies & Integration Risks

**E14 — Complex upstream dependency chain for SIP public exposure** 💡

SIP public exposure depends on three external systems: Cloudflare DNS (configured), Tailscale ACL (implied), and the VPS provider's NAT/filter policy. The fact that `tcpdump` shows zero SYN packets reaching the host indicates the block is upstream, not on the host. This is a critical path risk with no internal mitigation.

> **Suggestion:** Document the exact upstream dependency chain in the runbook with contact/escalation paths for each provider. Create a checklist for validating each layer (DNS resolution → SYN reachability → SIP response) before declaring Stage 6 complete.

## Cross-Lens Synthesis

### X1 — Public SIP exposure is blocked by an upstream dependency, not code 🎯

**Convergence:** Product readiness (Stage 6) and engineering readiness both depend on resolving an external network policy issue. The code is ready; the infrastructure is not.

**Insight:** This is a classic "last mile" risk. The project should treat upstream SIP exposure as a critical-path project management task, not just an ops task. Assign an owner, set a deadline, and define a fallback (e.g., alternative provider or VPN tunnel).

### X2 — `:latest` tag usage undermines both operational trust and product stability 🎯

**Convergence:** From the product side, a bad `:latest` push could take the SIP edge down without warning. From engineering, it violates the constitution and makes root-cause analysis harder.

**Insight:** Pinning images is usually a "nice to have," but for a SIP edge proxy where downtime means missed calls, it is essential. Bump Stage 3 (supply chain determinism) to run in parallel with Stage 1.

### X3 — Missing monitoring on VPS creates a product-engineering blind spot 🎯

**Convergence:** The product cannot claim "production-ready" status if operators cannot observe call volume, error rates, or backend health. Engineering has built the tools but disabled them due to resource constraints.

**Insight:** The monitoring overlay (Stage 5) should be treated as a hard dependency for public SIP exposure (Stage 6), not a follow-up. Reorder: monitoring overlay before public exposure.

### X4 — Audit log retention gap is both a legal risk and an engineering performance risk 🎯

**Convergence:** LGPD (product/compliance) requires data retention controls. Engineering notes unbounded table growth.

**Insight:** Implementing a 90-day audit log purge is a small engineering task with large product/compliance payoff. Do this before public exposure.

## Verdict

⚠️ **PROCEED WITH UPDATES**

TSiSIP has a strong architectural foundation and has successfully delivered an impressive 23+ feature set. The project should proceed toward production maturity, but **three actions must happen before public SIP exposure:**

1. **Pin image digests** (replace `:latest` in VPS compose).
2. **Enable monitoring/alerting** on the VPS (even a minimal overlay).
3. **Implement audit log retention purge** (90-day minimum).

Additionally, **reorder the roadmap** to parallelize public SIP exposure (Stage 6) with internal hygiene (Stages 1–3), and make monitoring (Stage 5) a hard dependency for Stage 6.

The engineering architecture is sound. The remaining work is operational hardening, not architectural rework.

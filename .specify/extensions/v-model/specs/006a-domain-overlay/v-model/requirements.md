# V-Model Requirements Specification: Domain Overlay Architecture


**Feature Branch**: `feature/006a-domain-overlay`
**Created**: 2025-07-19
**Status**: Draft
**Source**: `specs/006a-domain-overlay/spec.md`

## Overview

This document formalizes the requirements for introducing a file-based domain overlay architecture to the spec-kit-v-model extension. The overlay architecture cleanly separates base (domain-agnostic) command and template content from domain-specific safety/regulatory enrichment layers. A single `domain` field in `v-model-config.yml` controls all domain behavior. When set, overlay content is appended after base content using additive composition. When absent or empty, only base content is used — no domain-specific language appears. The feature refactors 9 contaminated command files (6 MIXED, 3 HARDCODED), 7 gated template files, and 9 extension.yml command descriptions to achieve clean base content that references only universally applicable standards (IEEE 1016, ISO 29119, IEEE 42010, INCOSE). Three initial domain overlays are created: `iso_26262` (automotive), `do_178c` (aviation), and `iec_62304` (medical devices). Because this feature modifies the content of 9 existing commands, the V-Model artifacts of their parent features (001 through 005) must be evolved using the ID lifecycle model defined in Feature 006b — items whose content changes are marked MODIFIED, and their downstream V-Model artifacts become SUSPECT until reviewed and resolved. All requirements are atomized from the feature specification using the strict translator constraint.

## Requirements

### Functional Requirements

| ID | Description | Priority | Rationale | Verification Method |
|----|-------------|----------|-----------|---------------------|
| REQ-001 | The extension SHALL provide a `commands/overlays/` directory where each subdirectory is named with a domain ID and contains command overlay files that provide domain-specific instruction content for the corresponding base command. | P1 | The overlay architecture requires a filesystem structure to separate domain-specific command content from base commands. | Inspection |
| REQ-002 | The extension SHALL provide a `templates/overlays/` directory where each subdirectory is named with a domain ID and contains template overlay files that provide domain-specific output sections for the corresponding base template. | P1 | Template content must be separated symmetrically with commands so that both instruction content and output structure support domain enrichment. | Inspection |
| REQ-003 | Each domain overlay directory (under both `commands/overlays/` and `templates/overlays/`) SHALL contain a `_domain.yml` manifest file that declares the domain's display name, governing standards, classification system name, and the list of commands that have overlay content in that domain. | P1 | The manifest enables tooling to validate overlay completeness, display domain metadata, and enumerate available overlays without scanning the filesystem. | Inspection |
| REQ-004 | The `config-template.yml` SHALL include a commented-out `domain` field with inline documentation listing the supported values (`iso_26262`, `do_178c`, `iec_62304`) and stating that omitting or leaving it empty activates domain-agnostic mode. | P1 | Users currently have no documented way to configure domain behavior; the existing conditional checks in 6 commands reference a field that does not exist in the config template. | Inspection |
| REQ-005 | When `domain` is set in `v-model-config.yml` and a command overlay file exists at `commands/overlays/{domain}/{command-name}.md`, the overlay content SHALL be appended after the base command instructions during command execution. | P1 | Additive composition ensures domain enrichment extends (not replaces) the base command, so the base experience is consistent regardless of domain. | Test |
| REQ-006 | When `domain` is set in `v-model-config.yml` and a template overlay file exists at `templates/overlays/{domain}/{template-name}-overlay.md`, the overlay template content SHALL be appended after the base template content. | P1 | Template overlays provide domain-specific output sections (e.g., ASIL tables, MC/DC coverage) that extend the base output structure. | Test |
| REQ-007 | When `domain` is not set, empty, or `v-model-config.yml` does not exist, every command and template SHALL use only its base content with zero domain-specific language, sections, or terminology in the output. | P1 | Zero-config default: teams that do not work in a regulated domain must never encounter safety standard jargon (ASIL, DAL, MC/DC, SIL/HIL, MISRA, CERT-C). | Test |
| REQ-008 | When `domain` is set but no overlay file exists for a specific command or template, the base content SHALL be used without generating an error, warning, or user-visible notification. | P1 | Graceful fallback: not every command requires domain enrichment; missing overlays are expected and must not disrupt the workflow. | Test |
| REQ-009 | Every command currently classified as MIXED (`system-design.md`, `system-test.md`, `architecture-design.md`, `integration-test.md`, `module-design.md`, `unit-test.md`) SHALL have all safety-standard references, domain-specific section headers, and domain-specific table columns extracted from the base file and placed into overlay files for each applicable domain. | P1 | The 6 MIXED commands have conditional checks but still leak domain terminology in section headers, table fields, and descriptions visible to the user even when domain is not configured. | Inspection |
| REQ-010 | The base `trace.md` command SHALL have the term "regulatory-grade" replaced with domain-agnostic language, all unconditional regulatory standard references (DO-178C, ISO 26262, IEC 62304, FDA 21 CFR 820, IEC 61508) removed from the Goal and Operating Constraints sections, and a domain overlay created containing the regulatory compliance framing. | P1 | `trace.md` is classified HARDCODED: it unconditionally references 5 safety standards with no domain check, making the traceability command appear safety-only even for general-purpose projects. | Inspection |
| REQ-011 | The base `hazard-analysis.md` command SHALL have "ISO 14971/ISO 26262-compliant" removed from its description, domain-specific severity scales (ISO 26262 ASIL, DO-178C Failure Conditions) moved to domain overlay files, and a general-purpose FMEA framing retained as the default base behavior. | P1 | `hazard-analysis.md` is classified HARDCODED: its description and severity tables reference domain-specific standards unconditionally, though FMEA methodology is applicable to non-regulated projects when framed generically. | Inspection |
| REQ-012 | The base `peer-review.md` command SHALL have the Governing Standard mapping table parameterized so that safety-standard names (DO-178C, ISO 26262, ISO 14971) appear only when a domain is configured, with domain-agnostic standard names (IEEE 1016, ISO 29119, INCOSE) used as defaults. | P1 | `peer-review.md` is classified HARDCODED: its governing standard table and section headers unconditionally embed safety standard names with no domain check. | Inspection |
| REQ-013 | After refactoring, every base command file SHALL reference only universally applicable standards (IEEE 1016, ISO 29119, ISO 29119-4, IEEE 42010, INCOSE) and SHALL NOT contain any of the following domain-specific terms: ASIL, DAL, SIL, HIL, MC/DC, WCET, MISRA, CERT-C, FMEA severity scales tied to specific standards, or "regulatory-grade." | P1 | The base experience must be genuinely clean; any domain-specific terminology remaining in base commands defeats the purpose of the overlay architecture. | Inspection |
| REQ-014 | Every base template currently classified as GATED (`architecture-design-template.md`, `hazard-analysis-template.md`, `integration-test-template.md`, `module-design-template.md`, `system-design-template.md`, `system-test-template.md`, `unit-test-template.md`) SHALL have its `<!-- SAFETY-CRITICAL SECTION -->` gated content extracted into domain overlay template files, leaving the base template with no commented-out safety sections. | P1 | Templates already use a clean gating pattern that serves as a migration guide; moving gated content to overlay files completes the separation so base templates contain no domain artifacts, even as comments. | Inspection |
| REQ-015 | The 9 command descriptions in `extension.yml` that currently reference safety standards unconditionally SHALL be updated to use domain-agnostic language while preserving the command's functional description. | P2 | Extension.yml descriptions are user-facing (displayed in the Copilot UI); they should describe what the command does generically, not assume a safety-critical context. | Inspection |
| REQ-016 | The `safety-critical` tag in `extension.yml` SHALL be retained without modification. | P2 | The tag describes the extension's capability (it CAN be used for safety-critical projects), not a hardcoded requirement; removing it would reduce discoverability for the target audience. | Inspection |
| REQ-017 | The extension SHALL provide an `iso_26262` domain overlay containing command and template overlay files for all commands where ISO 26262-specific content was extracted from the base, including at minimum: system-design, system-test, architecture-design, integration-test, module-design, unit-test, trace, hazard-analysis, and peer-review. | P1 | ISO 26262 (automotive functional safety) is the most referenced standard in the current codebase; its overlay must be complete to avoid regression for automotive teams. | Inspection |
| REQ-018 | The extension SHALL provide a `do_178c` domain overlay containing command and template overlay files for all commands where DO-178C-specific content was extracted from the base, including at minimum: architecture-design, module-design, unit-test, trace, hazard-analysis, and peer-review. | P1 | DO-178C (airborne software) is the second most referenced standard; its overlay ensures aviation teams get domain-appropriate content. | Inspection |
| REQ-019 | The extension SHALL provide an `iec_62304` domain overlay containing command and template overlay files for all commands where IEC 62304-specific content was extracted from the base, including at minimum: hazard-analysis, trace, and peer-review. | P1 | IEC 62304 (medical device software) is referenced in hazard-analysis and trace; its overlay provides medical-specific framing. | Inspection |
| REQ-020 | Each command that supports domain overlay enrichment SHALL include a standardized domain loading instruction block that replaces any existing ad-hoc conditional patterns, with the instruction: "Load v-model-config.yml. If `domain` is set and an overlay exists at `commands/overlays/{domain}/{command-name}.md`, append its content after these base instructions." | P1 | The 6 MIXED commands use inconsistent ad-hoc conditionals; replacing them with a single standardized pattern ensures consistent domain loading behavior across all commands. | Inspection |
| REQ-021 | Overlay command content SHALL use preference-based indirection (e.g., "prefer the domain's severity scale over the general-purpose scale") rather than duplicating base command content. | P2 | Preference-based language keeps overlays focused on domain-specific additions and avoids content duplication that would create maintenance burden when base commands evolve. | Inspection |

### Cross-Feature Lifecycle Requirements

| ID | Description | Priority | Rationale | Verification Method |
|----|-------------|----------|-----------|---------------------|
| REQ-LC-001 | When a base command file is refactored to extract domain-specific content, the V-Model artifacts of the parent feature that originally specified that command's content SHALL be evolved using the ID lifecycle model from Feature 006b: items whose content changes SHALL be marked MODIFIED, and their immediate downstream V-Model artifacts SHALL be marked SUSPECT. | P1 | Refactoring 9 commands changes the content specified in Features 001–005. Without lifecycle tracking, the parent specifications silently become stale — auditors and future contributors see spec IDs that no longer match the implementation. | Inspection |
| REQ-LC-002 | For each MIXED command (`system-design.md`, `system-test.md`, `architecture-design.md`, `integration-test.md`, `module-design.md`, `unit-test.md`), the corresponding parent feature spec SHALL have the affected requirement and design IDs marked MODIFIED with the rationale "Domain-specific content extracted to overlay per Feature 006a." | P1 | The 6 MIXED commands span Features 002 (system-design, system-test), 003 (architecture-design, integration-test), and 004 (module-design, unit-test). Their feature specs contain requirement IDs (REQ-NNN) and design IDs (SYS-NNN, ARCH-NNN, MOD-NNN) that describe content now moving to overlays. | Inspection |
| REQ-LC-003 | For each HARDCODED command (`trace.md`, `hazard-analysis.md`, `peer-review.md`), the corresponding parent feature spec SHALL have the affected requirement and design IDs marked MODIFIED with the rationale "Unconditional domain-specific content removed from base and relocated to overlay per Feature 006a." | P1 | The 3 HARDCODED commands belong to Features 001 (trace), 005a (hazard-analysis), and 005c (peer-review). Their parent specs describe domain-specific behavior that is now conditional on domain configuration. | Inspection |
| REQ-LC-004 | After marking parent feature IDs as MODIFIED, all downstream V-Model artifacts that trace to those IDs (acceptance plans, test plans, design documents) SHALL be marked SUSPECT and SHALL be resolved through re-running the corresponding V-Model commands with human review, following the cascade protocol defined in Feature 006b. | P1 | The lifecycle model requires that modifications cascade: a MODIFIED REQ makes its downstream ATP, SYS, STP, etc. SUSPECT. Each suspect must be reviewed and resolved (confirmed, updated, or deprecated) to maintain traceability integrity. | Inspection |
| REQ-LC-005 | When a parent feature specification item is marked MODIFIED solely because domain-specific content was extracted to an overlay (not because the item's functional intent changed), the SUSPECT downstream items MAY be resolved by confirming them as still valid without content changes, provided the functional behavior described by those items has not changed. | P2 | Not every modification requires rework. If extracting "ASIL Decomposition" from system-design leaves the base system design functionally identical, downstream system-test items that test the base behavior remain valid. The lifecycle protocol allows "confirm still valid" as a resolution. | Inspection |

### Non-Functional Requirements

| ID | Description | Priority | Rationale | Verification Method |
|----|-------------|----------|-----------|---------------------|
| REQ-NF-001 | The base content produced by any command when `domain` is not configured SHALL be identical to the base content produced when `domain` is configured — domain selection SHALL NOT alter the base portion of any output. | P1 | Base immutability guarantees that domain configuration is purely additive; non-regulated teams get the exact same experience whether or not overlay files exist in the extension. | Test |
| REQ-NF-002 | Domain IDs SHALL use snake_case format (e.g., `iso_26262`, `do_178c`, `iec_62304`) and SHALL be used consistently across directory names, `_domain.yml` references, and the `domain` field in `v-model-config.yml`. | P1 | Consistent, filesystem-safe naming avoids cross-platform path issues and matches the format already used in the 6 existing conditional checks. | Inspection |

### Constraint Requirements

| ID | Description | Priority | Rationale | Verification Method |
|----|-------------|----------|-----------|---------------------|
| REQ-CN-001 | This feature SHALL NOT modify, add, or remove any scripts (Bash or PowerShell), validation logic, or CLI flags — it restructures content (instructions, prompts, templates) only. | P1 | The spec explicitly states: "This feature restructures content, not runtime behavior." | Inspection |
| REQ-CN-002 | This feature SHALL NOT add or remove any commands — the 14 existing commands SHALL remain, including `hazard-analysis` which SHALL remain available without a domain configured using a general-purpose FMEA framing. | P1 | The spec explicitly states: "No commands are added or removed." | Inspection |
| REQ-CN-003 | A project SHALL support only one active domain at a time as specified by the single `domain` field in `v-model-config.yml`. | P1 | The spec explicitly excludes multi-domain support: "A project can only have one active domain at a time." | Inspection |
| REQ-CN-004 | The user SHALL manually set the `domain` field in `v-model-config.yml` — there SHALL be no auto-detection of domain from project content, repository metadata, or file analysis. | P1 | The spec explicitly excludes automated overlay selection to keep the mechanism simple and predictable. | Inspection |
| REQ-CN-005 | Only three domain overlays SHALL be created in this feature: `iso_26262`, `do_178c`, and `iec_62304`. | P2 | The spec explicitly scopes to three initial domains; additional domains follow the established pattern in future features. | Inspection |

## Assumptions

- The Copilot Extensions runtime supports appending overlay file content to base command content when the overlay path is specified in the command's instruction flow.
- The `v-model-config.yml` file is accessible from the repository root at command execution time, consistent with how the 6 existing conditional checks reference it.
- Domain-specific content currently embedded in commands and templates is complete and correct — it will be extracted as-is to overlay files without rewriting or expanding the safety content itself.
- The `<!-- SAFETY-CRITICAL SECTION -->` HTML comment gating pattern in templates accurately delimits the content that must be extracted to overlay template files.
- Feature 006b (ID Lifecycle Model) is implemented concurrently or prior to the cross-feature evolution work, providing the lifecycle states (MODIFIED, SUSPECT) and resolution protocol needed by REQ-LC-001 through REQ-LC-005.

## Dependencies

- **Existing command and template files**: The 14 command files and 12 template files must exist in their current form as the source for extraction and refactoring.
- **`v-model-config.yml` support**: The extension runtime must support reading configuration from `v-model-config.yml` at the repository root (this capability already exists via the 6 commands that check the `domain` field).
- **Feature 006b (ID Lifecycle Model)**: The lifecycle states and cascade protocol defined in Feature 006b are required for the cross-feature evolution requirements (REQ-LC-001 through REQ-LC-005). While the overlay infrastructure itself (REQ-001 through REQ-021) can be implemented independently, evolving the parent feature specifications requires 006b's lifecycle mechanism.

## Glossary

| Term | Definition |
|------|-----------|
| Base content | The domain-agnostic command or template content that is used regardless of domain configuration; references only universally applicable standards |
| Overlay | A domain-specific file that extends base content with additional instructions, sections, or template structures when the corresponding domain is configured |
| Domain ID | A snake_case identifier for a safety/regulatory domain (e.g., `iso_26262`, `do_178c`, `iec_62304`) used in directory names and configuration |
| `_domain.yml` | A manifest file in each domain overlay directory declaring the domain's metadata: display name, governing standards, classification system, and covered commands |
| Additive composition | The overlay architecture's composition model where domain overlay content is appended after base content, never replacing or modifying the base |
| Graceful fallback | The behavior where a missing overlay file for a configured domain results in using the base content without error or warning |
| MIXED command | A command that has domain conditional checks but still leaks domain-specific terminology in visible text (section headers, descriptions, table columns) |
| HARDCODED command | A command that unconditionally references domain-specific standards with no conditional checks |
| GATED template | A template where domain-specific content is enclosed in `<!-- SAFETY-CRITICAL SECTION -->` HTML comments, already structurally separated but still inline |
| Preference-based indirection | An overlay instruction pattern that says "prefer domain content over base content" rather than duplicating or replacing the base |
| MODIFIED (lifecycle) | An ID lifecycle state from Feature 006b indicating that an item's content has been updated in-place while preserving its ID; downstream items become SUSPECT |
| SUSPECT (lifecycle) | An ID lifecycle state from Feature 006b indicating that a parent item was modified or deprecated; the suspect item needs review to confirm, update, or deprecate |

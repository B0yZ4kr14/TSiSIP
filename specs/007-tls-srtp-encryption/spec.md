# Feature Specification: End-to-End Encryption for SIP Signaling and Media

## Overview

| Field | Value |
|-------|-------|
| **Feature** | End-to-End Encryption for SIP Signaling (TLS) and Media (SRTP) |
| **Short name** | tls-srtp-encryption |
| **Created** | 2026-05-16 |
| **Status** | In Progress (53% — TLS listener done; mTLS trunk, SRTP, cipher hardening, rotation monitoring pending) |
| **Last Updated** | 2026-05-19 |
| **Context** | TSiSIP handles sensitive VoIP traffic. SIP signaling carries authentication credentials and call metadata; RTP carries voice content. Encryption is required to prevent eavesdropping, toll fraud, and identity spoofing. |
| **Objective** | Deliver mutual TLS authentication for SIP trunks, a self-signed CA infrastructure, zero-downtime certificate rotation, SRTP key exchange via SDP, and hardened cipher suites across the TSiSIP SIP edge and RTPengine stack. |

## User Scenarios & Testing

### Scenario 1: SIP Trunk Registers via Mutual TLS
**Given** a carrier trunk is configured with a client certificate signed by the TSiSIP CA,
**When** the trunk sends a REGISTER over TLS to OpenSIPS on 5061/tcp,
**Then** OpenSIPS verifies the client certificate against the CA, authenticates the trunk, and permits the registration.

### Scenario 2: Certificate Rotation Without Downtime
**Given** the TSiSIP CA certificate is nearing expiry,
**When** the operator deploys a new CA and server certificate into the OpenSIPS container via Docker secrets,
**Then** OpenSIPS reloads the TLS profile via MI `tls_reload` without dropping existing TLS connections or restarting the process.

### Scenario 3: SRTP Established for a TLS-Signaled Call
**Given** a UA negotiates a call through OpenSIPS over TLS,
**When** the INVITE/200 OK SDP passes through RTPengine with `RTP/SAVP` or `UDP/TLS/RTP/SAVP` (DTLS-SRTP),
**Then** RTPengine negotiates SRTP keys, encrypts the media stream, and reports the SRTP cipher in use.

### Edge Case 1: Downgrade Attack from TLS to UDP
An attacker strips TLS and forces the call over unencrypted UDP. OpenSIPS must reject the request or require encryption for trunk-facing interfaces.

### Edge Case 2: Expired Client Certificate on Trunk
A carrier trunk presents an expired certificate. OpenSIPS must reject the TLS handshake with `alert_certificate_expired` and not fall back to TCP/UDP.

### Edge Case 3: SRTP Key Mismatch After Re-INVITE
A session refresh or hold/re-invite generates new SDP. RTPengine must update the SRTP context without media dropout or key desynchronization.

## Functional Requirements

### FR-007-001: Mutual TLS Authentication for SIP Trunks
- OpenSIPS `tls_mgm` module configures a TLS profile requiring `verify_client = 1` (mandatory client cert) on the trunk listener (5061/tcp).
- Client certificates are signed by the project-owned CA; revoked certs are tracked via a CRL file.
- **Acceptance Criteria**: Trunk registration with valid client cert succeeds; registration without client cert fails at TLS handshake; packet capture shows no plaintext SIP on 5061.

### FR-007-002: Self-Signed CA Infrastructure
- A `ca_tool` container or offline script generates the root CA, intermediate CA, server certificate for OpenSIPS, and client certificates for trunks.
- Key length: RSA 4096 or ECDSA P-256; hash algorithm: SHA-256.
- CA certificate and CRL are distributed to OpenSIPS via Docker secrets.
- **Acceptance Criteria**: `openssl verify -CAfile ca.crt server.crt` returns `OK`; `openssl s_client -connect` shows chain depth 2.

### FR-007-003: Certificate Rotation Without Downtime
- New certificates are staged as Docker secrets (e.g., `server_cert_v2`).
- OpenSIPS MI command `tls_reload` loads the new certificate without process restart.
- Existing TLS connections continue using the old certificate until natural closure; new connections use the new certificate.
- **Acceptance Criteria**: `tls_reload` executes in <1s; active TLS call remains stable; new TLS handshake uses new certificate.

### FR-007-004: SRTP Key Exchange via SDP
- RTPengine `offer`/`answer` commands include `transport-protocol=RTP/SAVP` or `UDP/TLS/RTP/SAVP`.
- SRTP keys are generated by RTPengine and inserted into SDP; OpenSIPS relays the SDP without inspecting keys.
- **Acceptance Criteria**: SDP `a=crypto` lines are present in both directions; RTPengine log confirms `SRTP encryption activated`.

### FR-007-005: Cipher Suite Hardening
- OpenSIPS TLS profile disables TLS 1.0/1.1; minimum version is TLS 1.2; TLS 1.3 is preferred.
- Allowed cipher suites: `ECDHE-ECDSA-AES256-GCM-SHA384`, `ECDHE-RSA-AES256-GCM-SHA384`, `TLS_AES_256_GCM_SHA384`.
- RTPengine SRTP cipher preference: `AES_256_CM_HMAC_SHA1_80` or `AES_GCM_256`.
- **Acceptance Criteria**: `openssl s_client -tls1_3` succeeds; `openssl s_client -tls1_1` fails; RTPengine reports only allowed SRTP ciphers.

| Entity | Description | Attributes |

## Scope

### In Scope
- OpenSIPS `tls_mgm` and `proto_tls` module configuration.
- Self-signed CA generation, server certificate issuance, and client certificate distribution.
- Docker secrets integration for certificate storage.
- MI-driven `tls_reload` for zero-downtime rotation.
- RTPengine SRTP negotiation via SDP (`RTP/SAVP`, DTLS-SRTP).
- Cipher suite allowlists and TLS version enforcement.

### Out of Scope
- Public CA-issued certificates (e.g., Let's Encrypt) for SIP TLS.
- ZRTP (media encryption negotiated directly between UAs without SDP).
- S/MIME encryption of SIP message bodies.
- IPsec or VPN tunneling as an alternative to TLS.
- Certificate pinning inside SIP UAs.

## Dependencies

| Dependency | Description | Impact if Missing |
|------------|-------------|-------------------|
| OpenSIPS 3.6 LTS | `tls_mgm`, `proto_tls` modules | Cannot terminate TLS or enforce mTLS |
| RTPengine | SRTP kernel module and userspace daemon | Cannot encrypt media streams |
| OpenSSL ≥ 1.1.1 | TLS 1.3 and modern cipher support | Weak TLS and no TLS 1.3 |
| Docker Secrets | Secure certificate distribution at runtime | Certificates leak into image layers |

## Assumptions

- Trunk operators can generate and install client certificates signed by the TSiSIP CA.
- The TLS listener uses a dedicated port (5061/tcp) to avoid co-mingling with unencrypted TCP (5060/tcp).
- RTPengine is compiled with SRTP and OpenSSL support.
- Certificate rotation is a planned, infrequent event (e.g., annually or on compromise).

## Risks

| ID | Risk | Likelihood | Impact | Mitigation |
|----|------|------------|--------|------------|
| R-001 | Client certificate management overhead discourages trunk onboarding | Medium | Medium | Provide automated cert issuance script and clear documentation |
| R-002 | tls_reload drops existing connections on some OpenSIPS builds | Low | High | Test thoroughly in staging; plan maintenance window as fallback |
| R-003 | SRTP CPU overhead degrades call capacity on RTPengine | Medium | Medium | Benchmark with SRTP on; consider dedicated RTPengine hosts |
| R-004 | Weak legacy UA cannot negotiate hardened cipher suite | Medium | Medium | Maintain a separate legacy TLS profile with weaker ciphers on isolated listener (only if absolutely required) |
| R-005 | CA private key compromise invalidates entire trust chain | Low | Critical | Store CA key offline or in HSM; maintain revocation list; plan emergency re-issuance |

## Notes

- OpenSIPS is the only public SIP endpoint; TLS termination happens only in OpenSIPS, not in a separate reverse proxy.
> **Constitution Reference**: See `.specify/memory/constitution.md` §2 — PostgreSQL is the sole persistence layer; `db_postgres` is the only OpenSIPS DB module.
> **Constitution Reference**: See `.specify/memory/constitution.md` §3 — The `sanity` module is forbidden in OpenSIPS 3.6 LTS.
- All certificate files and keys must be injected via Docker secrets or mounted read-only volumes; never commit them to Git.
- Service and network names use lowercase snake_case (e.g., `sip_edge`, `sip_internal`).
- Backend RTP addresses must not leak externally; SRTP ensures media confidentiality even if addresses are discovered.
